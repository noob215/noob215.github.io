代码审计file_get_contents使用不当造成ssrf

前言

题目来自PHP SECURITY CALENDAR 2017

Day 20 - Stocking

题目叫作Stocking

漏洞解析

这一关考察的是file_get_contents()以及第25行的报错信息造成的内网信息泄露

分析一下程序,程序接收$_GET[‘img’],然后传到ImageLoader类中的getResult方法中,可以看到它进行了过滤filter_var($uri, FILTER_VALIDATE_URL)实际上这个过滤对http(s)://和file:///是没有用的，19行又对请求的文件头属性进行了检测如果不是jpg的属性头，所以没法直接进行一个读取,但是通过25行的报错,我们可以获得一些信息。

举例

当我尝试读取本地的22端口时因为报错会爆出一些信息

我在D:\phpstudy下放了一个1.txt

当我尝试读取时他显示的是only .jpg files allowed 这说明file协议他也不会被过滤的

所以我们可以对一些使用file_get_contents()没过滤完全的程序利用http(s)和file协议进行ssrf,利用一般为schema://localhost:port或者schema://127.0.0.1:port,其实这里还可以利用dict和gophar等协议在这里不做太多讨论

实例分析

这个例子采用的是NoneCMS1.3,目录结构是这样的采用了thinkphp的框架开发,开始全局搜索file_get_contents()可以看到在/noneCms-1.3.0/vendor/electrolinux/phpquery/phpQuery/phpQuery.php的第409行调用了该函数

全局搜索哪里调用了newDocumentFile()可以看到在D:/phpstudy/WWW/noneCms-1.3.0/noneCms-1.3.0/application/admin/controller/Article.php第183行调用了newDocumentFile()所以此处可以作为一个利用点

我们找到该出的利用点是后台管理的转载文章处然后看一下Article.php处的代码的利用条件
条件绕过都很简单,不过因为这个ssrf是没有回显的所以我请求了一下ceye.io

虽然它会显示系统发生错误但是，可以看到还是请求成功了payload**url=http%3A%2F%2F98avgk.ceye.io/%3Fcsdn%3D1&cid=1**

不过因为这个ssrf是没有回显的所以可以说是一个比较鸡肋的洞,但是这就是一个file_get_contents错误使用的结果

修复建议

建议在开发的时候file_get_contents之后,加上对请求的地址的过滤,以及请求文件的类型判断,不要用简单的字符串位置判断结果,建议使用白名单的方式来进行防御
